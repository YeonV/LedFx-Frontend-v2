## 🗺️ Project Topology

This is a distributed, multi-repo ecosystem. You are currently operating inside the **Frontend** repository.

### Core Architecture
- **Backend (Python):** https://github.com/LedFx/LedFx
- **Frontend (React):** https://github.com/YeonV/LedFx-Frontend-v2
- **Audio-Visualiser (React Module):** https://github.com/YeonV/audio-visualiser
- **Dynamic Module Loader:** https://github.com/YeonV/react-dynamic-module (Custom package by YeonV used to link Frontend & Visualiser)
- **Android Bootstrap (Java/WebView):** https://github.com/YeonV/python-for-android (`develop` branch) — forked p4a; the webview bootstrap (`bootstraps/webview/`) is the only Java layer to edit for Android native behaviour
- **Android Build Recipes:** https://github.com/YeonV/ledfx-android — contains `buildozer.spec`, `src/main.py`, `src/service.py`, and the `deps/ledfx` submodule (pinned backend)

### Pipelines & Distribution
- **Desktop Build Pipeline:** https://github.com/YeonV/LedFx-Builds
  - *Trigger Flow:* `Frontend/release.yml` → `LedFx-Builds/BuildPipelineUV.yml` + `BuildAndroidMatrix.yml`
  - *Binaries:* Published to LedFx-Builds/releases
- **Android Build Pipeline:** https://github.com/YeonV/ledfx-android (`.github/workflows/build-apk.yml`)
  - *Trigger Flow (debug):* any push to `main` → artifact APK uploaded to workflow run
  - *Trigger Flow (release):* push a version tag (e.g. `v2.1.4`) → GitHub Release created with APK
  - *p4a source:* fetched at build time from `p4a.fork = YeonV` + `p4a.branch = develop` — commit to `_python-for-android` first, then trigger the Android build
  - ⚠️ **Cannot build locally** — no Android SDK/NDK/Java on dev machine; always use GitHub Actions

### Android App Architecture
The Android app (`com.ledfx.ledfx`) has a specific two-process runtime structure:
- **UI Process** (`com.ledfx.ledfx`): WebView running the React frontend + `org.kivy.android.PythonActivity`
- **Service Process** (`com.ledfx.ledfx:service_ledfx`): LedFx Python backend running as a foreground sticky service

**Service naming convention:** `buildozer.spec` declares `services = ledfx:service.py:foreground:sticky`. p4a generates a Java class named `Service<Name|capitalize>` in the app package → `com.ledfx.ledfx.ServiceLedfx`. Any Java code that starts or stops the service must use this exact class name, NOT the base `org.kivy.android.PythonService`.

**Build chain:**
```
buildozer.spec  (p4a.fork = YeonV, p4a.branch = develop)
    → _python-for-android/pythonforandroid/bootstraps/webview/   ← edit Java here
    → generates ServiceLedfx.java from Service.tmpl.java
    → AndroidManifest registers com.ledfx.ledfx.ServiceLedfx
```

**ADB debugging (device must be connected via USB with USB debugging enabled):**
```bash
adb devices                                          # confirm device is listed
adb shell pm list packages | grep ledfx              # find package name
adb shell dumpsys activity services com.ledfx.ledfx  # inspect running services
adb shell ps -A | grep ledfx                         # confirm process separation
adb logcat -s PythonActivity:W python:I ledfx:I      # live logs
```

**⚠️ CRITICAL: Android APK cannot be built locally.** Do NOT attempt `buildozer android debug` on the dev machine — it requires a full Android SDK/NDK/Java toolchain that is intentionally not installed. All Android builds must go through GitHub Actions.

**Triggering an Android build:**
- **Debug build (artifact only):** push any commit to `_pipeline_android` (`main` branch) → `build-apk.yml` runs automatically on `on: push`
- **Release build (creates GitHub Release):** push a version tag to `_pipeline_android` → e.g. `git tag v2.1.4 && git push origin v2.1.4`
- The workflow fetches `_python-for-android` at build time via `p4a.fork = YeonV` + `p4a.branch = develop` — so pushing to the p4a fork takes effect on the **next** triggered build, not automatically.

## ⚡ Electron (Desktop — LedFxCC)

The Electron main process lives **inside this repo** at `src/app/`. It is NOT a separate repository.

| File | Role |
|---|---|
| `src/app/electron.ts` | Entry point — BrowserWindow, tray, IPC setup |
| `src/app/app/handlers.mts` | `ipcMain` ↔ renderer bridge (all IPC channels) |
| `src/app/app/protocol.mts` | Deep-link handler (`ledfx://` protocol) |
| `src/app/app/utils/isCC.mts` | Detects whether backend binary is bundled |
| `src/app/app/utils/runCore.mts` | Spawns the bundled Python backend process |
| `src/app/app/utils/songDetector.mts` | Song Detector — **Electron-only feature** |

**`isCC()` behaviour:**
- Returns `true` → running as **LedFxCC** (bundled backend binary present) — self-contained desktop app
- Returns `false` → plain Electron shell connecting to an external backend

**Frontend-side IPC hooks** (consume `ipcRenderer` events inside React):
- `useIpcHandlers` — handles Electron IPC events
- `useProtocolHandler` — processes deep-link protocol messages
- `useDisplayMode` — OBS/fullscreen display mode (Electron-only)

**Commands:**

| Task | Command | Notes |
|---|---|---|
| Run in Electron (dev) | `yarn dev` | Hot-reloads React inside Electron window |
| Package as LedFxCC (local) | `yarn distcc` | Uses `electron-builder.json`; full CI packaging via `_pipeline/` |

## 🚀 Local Development Setup & Verification

To work on frontend features, you must clone and run both the Backend and Frontend. Open two separate terminal instances.

### Terminal 1: Backend (Port 8888)
1. `cd .. && git clone https://github.com/LedFx/LedFx.git`
2. `cd LedFx`
3. `uv install ledfx`
4. `uv run ledfx --offline -vv`

### Terminal 2: Frontend (Port 3000)
From the root of this frontend repository:
1. `yarn install`
2. `yarn start`

### 🔒 Strict Verification (REQUIRED)
Do not proceed with code changes until you verify both APIs are responding correctly.

- **Verify Backend:** Run `curl -s http://localhost:8888/api/info`
  - *Success:* Response JSON must contain `"name"` field with the string `"LedFx"`.
- **Verify Frontend:** Run `curl -s http://localhost:3000/manifest.json`
  - *Success:* Response JSON must contain `"name"` field with the string `"Blade"`.

## 🎨 Working with the Audio-Visualiser

The audio-visualiser is a standalone library compiled into a dynamic module (`yz-audio-visualiser.js`). It is injected into the Frontend at runtime.

If your task involves the visualiser, follow these steps:

### 1. Setup the Visualiser Workspace
1. `cd .. && git clone https://github.com/YeonV/audio-visualiser.git`
2. `cd audio-visualiser`
3. `pnpm i`
4. `pnpm dev` (Runs on Port `3001`)

### 2. Read the Visualiser Documentation
The Visualiser is highly complex and utilizes heavy auto-generation. Before modifying its logic, read the index map:
`cat ../audio-visualiser/docs/README.md`
*(Use this README to find specific docs on `ARCHITECTURE.md`, `ZUSTAND_STORE.md`, or `GENERATORS.md`.)*

### 3. The `postbuild` Gotcha (CRITICAL)
When you are ready to build the visualiser for the frontend, you must run `pnpm build`. 
**WARNING:** The `postbuild` script inside `audio-visualiser/package.json` copies `.js` and `.d.ts` files to `../frontend`. 
- **Action Required:** Before running `pnpm build`, check the `package.json` and modify the `postbuild` script path to exactly match the folder name of this Frontend repository (e.g., `../LedFx-Frontend-v2`) so the copy succeeds.

### 4. State Management Paradigm (Generation 5)
When making changes to how the Frontend communicates with the Visualiser, strictly adhere to the current Zustand paradigm:
- **`useStore`:** Global frontend state.
- **`useVstore`:** Direct access to the dynamically loaded Visualiser's internal state.
- **DO NOT** use prop drilling.
- **DO NOT** use `window.api` for state updates.
- **DO NOT** write polling intervals.
- The states are fully reactive. Read `../audio-visualiser/docs/COMMUNICATION.md` if you are confused.

## 🧪 Testing

We use Playwright for end-to-end testing to verify the frontend's integration with the Backend and Audio-Visualiser.

### 1. Preparation
Ensure all services are running before starting tests:
- **Backend:** Port 8888 (See Terminal 1 instructions above)
- **Frontend:** Port 3000 (See Terminal 2 instructions above)
- **Visualiser:** Port 3001 (Optional, required for visualiser tests)

### 2. Running Tests
- **Run all tests:** `yarn test:pw`
- **Run specific test:** `npx playwright test tests/initial-load.spec.ts`
- **Debug with UI:** `npx playwright test --ui`

### 3. Handling UI Overlays
The application features several modal overlays on initial load that must be handled in tests:
- **Host Selection:** Appears if no backend is remembered. Look for `http://localhost:8888`.
- **Setup Assistant (Intro):** Appears on first load. Click the **Skip** button to reach the dashboard.

### 4. Visualiser Development Testing
The visualiser tests rely on the module being served from port 3001. When testing the visualiser, ensure the dev server in `audio-visualiser/` is active. Visualiser rendering often requires a safety delay (e.g., `page.waitForTimeout(5000)`) to allow the dynamic module to fully initialize.

### 5. Writing New Tests
- **Location:** `tests/`
- **Naming:** Use `.spec.ts` suffix.
- **Verification:** Always verify UI state changes and capture screenshots for visual inspection using `page.screenshot()`.
